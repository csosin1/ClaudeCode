# Lessons Learned

## 2026-04-18 SQLite write-write contention crashes pipelines without busy_timeout
- **What went wrong:** `overnight_post_markov.sh`'s carmax export crashed at `migrate_dist_date_iso` (schema.py:31) with `sqlite3.OperationalError: database is locked`. The carmax dashboard.db was truncated to 0 bytes and live stayed on the prior (stale-cache) regen for ~50 minutes. Three writers were active on `carmax_abs.db` at the same moment: (a) the methodology pipeline trying to add+populate `dist_date_iso`; (b) the daily 14:17 UTC ingest cron (`run_ingestion.py`, processing summary tables for deal 2023-2 of 37); (c) `unified_markov.py` mid-checkpoint on the trajectory-builder run.
- **Root cause:** `sqlite3.connect()` in `*_abs/export_dashboard_db.py` was created with default settings — `timeout=0` and no `PRAGMA busy_timeout`. SQLite's WAL mode supports many readers + one writer, but a SECOND writer fails immediately rather than waiting. The methodology export and the cron ingest both wanted the write lock on `pool_performance` simultaneously, and whoever lost the race got the bare `database is locked` exception with no retry.
- **What to do differently:**
  - **Fix shipped:** both `carvana_abs/export_dashboard_db.py` and `carmax_abs/export_dashboard_db.py` now open their source connection with `timeout=300` and `PRAGMA busy_timeout=300000` (5 minutes). Under WAL the only thing we'd ever wait on is another writer, and those finish in seconds — letting SQLite wait is the right move; crashing is not.
  - **Preventive rule:** Any `sqlite3.connect()` against a multi-writer DB on this droplet (the source DBs in `carvana_abs/db/` and `carmax_abs/db/` qualify — both ingest cron + render cron + Markov touch them) MUST set a busy_timeout. Default-zero is a footgun.
  - **How to apply:** When introducing a new code path that opens one of the source DBs for writes, copy the `timeout=300 / PRAGMA busy_timeout=300000` pattern. For a new long-running connection, also set `PRAGMA journal_mode=WAL` for symmetry.
  - **Orchestration follow-up:** the deeper issue is that two heavy pipelines (`overnight_post_markov.sh` and the daily `cron_ingest.sh`) can run concurrently and contend on the same DB. The methodology rebuild lock at `/opt/.methodology_rebuild.lock` (added in the same session) prevents auto-deploy from regenning during a rebuild, but does not prevent the daily cron from starting a heavy ingest. If contention recurs, escalate to a single shared "abs-source-db-writer" advisory lock — only one of {ingest, export, Markov-checkpoint} holds it at a time.

## 2026-04-18 Derived display columns escape DB-layer audits
- **What went wrong:** Two production bugs in 36 hours had the same root cause: columns computed at render time (not stored in DB) bypassed the data-audit-qa Phase 1-4 surface. Examples — 100× rendering of "Exp Loss" on the Residual Economics tab (caught by luck in spot-checks); WAL collapse to 0.1y on brand-new deals where only 1 distribution had arrived (caught by user, not audit). The DB values were fine; the display-layer math on top of them was wrong. Nothing in the ingestion / post-Markov / new-deal audit suite inspects rendered HTML, so derived columns had no automatic sanity gate.
- **Root cause:** Audit discipline stopped at the DB boundary. The renderer (`generate_dashboard.py`, `methodology_tab.py`, `generate_recent_trends_tab`) computes dozens of derived figures — percentage conversions, weighted averages, chained arithmetic across columns — and each of those computations was an unguarded code path. A single off-by-100 or stale-WAL assumption could ship to production and live there until a human noticed.
- **What to do differently:**
  - **Preventive rule:** Every derived / computed display column MUST have a sanity range defined at the point of definition. `audit_display_ranges.py` holds the canonical `DISPLAY_RANGES` dict; a PR that adds a new numeric column to any tab WITHOUT a matching range entry is an incomplete change and should be rejected in review.
  - **How to apply:** When adding a column to any `generate_*` function in the renderer, simultaneously add the entry to `DISPLAY_RANGES` (canonical key, `(lo, hi, unit, why)` tuple) and — if relevant — an `AGGREGATE_CHECKS` or cross-column chain-check entry. Header strings that map to the key live in `HEADER_ALIASES` (or `TRANCHE_ALIASES` for context-sensitive ones).
  - **Automation:** `audit_display_ranges.py` runs after every regeneration (manual via `ssh prod-private '/opt/abs-venv/bin/python /opt/abs-dashboard/audit_display_ranges.py --live'`), after every promote in `deploy/cron_ingest.sh`, and in the QA harness. HALT findings trigger a notify.sh push; because the data is already live by that point, we don't roll back — we alert + log to `AUDIT_FINDINGS.md` for human triage.
  - **Principle:** DB-layer audit invariants (Phase 1-4) remain necessary but are no longer sufficient. The DB layer and the display layer are different surfaces, and each needs its own sanity gate.

## 2026-04-18 Deals silently missed when config.py wasn't updated after new issuance
- **What went wrong:** 5 deals (Carvana 2021-P3, 2021-P4, 2025-P1, 2026-P1; CarMax 2026-2) had been issued by SEC EDGAR but were never added to `carvana_abs/config.py` / `carmax_abs/config.py` DEALS registries. Result: never ingested, never in Markov training, never in residual economics, never in any dashboard output. 4 of the 5 were actually tagged in a `# NOTE: Some deals are missing` TODO comment block in `carvana_abs/config.py` — the CIKs were explicitly called out as "to add later" but never actioned. The 5th (CarMax 2026-2) was issued after the last manual config refresh and no new-issuance watcher was wired up at the time.
- **Root cause:** The deal registry is the single source of truth for ingestion (`get_active_deals()` drives `run_ingestion.py`), so a missing entry = silent invisibility across the entire pipeline. There was no automated check that closed the loop between "SEC EDGAR has a new Carvana/CarMax trust CIK" and "DEALS dict has an entry for it." The TODO comment in `config.py` was a reminder to a human that never got picked up.
- **What to do differently:**
  - **Weekly discovery scanner is now live** at `deploy/discover_new_deals.py`, cron-installed via `deploy/cron.d/abs-discover-deals`. It queries EDGAR's EFTS full-text search for recent filings naming "Carvana Auto Receivables Trust" / "CarMax Auto Owner Trust", dedupes CIKs against the two DEALS dicts, and emits a notification with the exact edit needed. Runs weekly; user receives a notify.sh ping on any new CIK.
  - **Preventive rule:** TODOs/NOTE comments in `config.py` DEALS blocks are bug reports, not aspirations. Any time a DEAL registry comment says "missing"/"to add"/"can be discovered by searching directly," that is a regression by definition and must be fixed in the same change that noticed it, not deferred.
  - **Preventive rule:** Whenever a new data-source registry (`DEALS`, `SHELVES`, etc.) is introduced, ship the discovery/validation scanner alongside it — never rely on humans to notice the source universe has grown.
  - **Audit-side invariant:** Future audits should compare the DB's distinct-deal set against the count of CIKs SEC EDGAR returns for the issuer's full-text-search company filter, and flag if the DB is behind. Cheap check, catches this class of drift fast.
- **This session's fix:** All 5 deals added to their respective DEALS configs with CIKs verified against EDGAR submission JSON (name + SIC 6189 "Asset-Backed Securities"), ingested, parsed, scored through Markov, and included in the regenerated residual-economics tab.

## 2026-04-14 Non-negative amount regex silently dropped rows with negative monthly values
- **What went wrong:** CarMax parser's "Recoveries" and "Defaulted Receivables (charge-offs)" patterns used amount subpattern `\$\s*([\d,]+(?:\.\d+)?)` — no leading `-`. For 2018-2 and 2018-3 on Jan 2022 (the only 2 of 2,024 CarMax certs with net-negative monthly recoveries that period), the period row failed to match. With only the cumulative row matching, the single hit was stored as `recoveries` (22M — wrong), and `cumulative_liquidation_proceeds` was left NULL. Values on dashboard charts looked real but were in the wrong row.
- **Root cause:** Regex assumed domain values are always non-negative. Real-world financial time series have legitimate sign reversals (reversals/clawbacks, restatements). Silent mis-alignment, not a crash.
- **What to do differently:**
  - **Every amount regex that scrapes financial figures must accept optional leading `-`:** `\$\s*(-?[\d,]+(?:\.\d+)?)`. Same rule for parenthesized negatives `\(...\)` if the source uses those.
  - **If a regex uses `re.finditer` and you branch on `len(matches) > 1`:** when the count is off by one, the fallback path silently stores the wrong column into the wrong slot. Prefer explicit "first-occurrence-is-period, last-occurrence-is-cumulative" with a length assertion rather than `if len>1` that merges single-match cases into the period slot.
  - Preventive rule: post-ingest invariants should include "same-period recoveries ≡ liquidation_proceeds" (when both are monthly) — when they diverge by more than fp-noise the row is a candidate parser bug.

## 2026-04-14 Label-anchored patterns beat letter-prefix patterns for multi-vintage issuer certs
- **What went wrong:** CarMax parser used fixed letter prefixes (`a\.` / `b\.` / `c\.` / `d\.` / `e\.` / `f\.`) on delinquency buckets and the "Total Past Due" / "Delinquent Loans as a percentage" labels. 2014-2015 certs use 3 DQ buckets ending at letter `d.` (total); 2016+ certs use 4 buckets with total at `e.` and trigger % at `f.`. The fixed letters silently failed on 85 early-life rows of 2014-1 through 2015-4 (columns NULL; no error).
- **Root cause:** Letter prefix happens to be stable in any single vintage but **drifts across vintages** as field counts change. Anchoring on letter is anchoring on structure, not semantics.
- **What to do differently:**
  - Anchor on the **label body** (the English text of the field name), not the list-letter prefix. Use `[a-z]\.\s+<label body>` as the anchor, never `a\.\s+<body>`.
  - When issuer formats change shape (buckets added/split), a single pattern with a label-body alternation (`91 to 120|91 or more`) handles all vintages in one code path — no if-vintage-then-else branches.
  - Preventive rule: if ≥5% of rows have a NULL in a column that the cert source always populates for some vintage range, treat as "parser vintage gap" and search for letter-prefix or line-number anchors in the regex.

## 2026-04-14 `ORDER BY distribution_date` sorted lex — silently stale "latest" values
- **What went wrong:** `pool_performance.distribution_date` is stored as `M/D/YYYY` / `MM/DD/YYYY` text. Every SQL `ORDER BY distribution_date DESC LIMIT 1` returned a row picked by lex order, not chronological (`'9/12/2022' > '9/10/2025'` lex). "Current WAC", "Initial WAC", "Current CoD" summary fields, and any "latest pool row" lookup silently showed stale data on 14/16 Carvana deals and 47/49 CarMax deals. Invisible because the returned value was real — just from the wrong period (often years off).
- **Root cause:** Text storage in a `%m/%d/%Y` format that isn't lex-sortable. Fine for display, catastrophic for SQL sort. Python-side `.sort_values("period")` in some code paths masked it — but any direct `SELECT … ORDER BY distribution_date LIMIT 1` bypassed Python re-sorting and returned the lex-latest row.
- **What to do differently:**
  - Never sort date text unless the format is ISO-8601 (`YYYY-MM-DD`). If the raw column exists in a non-sortable format, add a parallel `*_iso` column via `ALTER TABLE`, backfill with `substr` + `printf('%02d', CAST(...))` in SQL, index it, and route `ORDER BY` to the ISO column.
  - Use a trigger, not parser changes, when ingestion must stay frozen. An `AFTER INSERT` trigger on the table auto-populates the ISO column on every write, with zero parser-code risk.
  - Preventive rule: `grep -rn 'ORDER BY [a-z_]*date' --include='*.py'` on any codebase where dates are stored as text; every hit that isn't ISO is suspect.
  - Test signal: if `SELECT … ORDER BY date_col DESC LIMIT 1` returns a different row than Python-parsed `max()`, you have a lex-sort bug.

## 2026-04-14 Python scripts in /tmp can load `/tmp/inspect.py` instead of stdlib
- **What went wrong:** A reparse script at `/tmp/cmx_reparse.py` failed at `from bs4 import BeautifulSoup` with `AttributeError: module 'inspect' has no attribute 'signature'`. ~15 min debugging an apparently impossible stdlib error.
- **Root cause:** CPython prepends the script's own directory to `sys.path[0]`. There was a stray `/tmp/inspect.py` from another project, which shadowed the standard library's `inspect` module when Python resolved imports during `typing_extensions` load. bs4 → typing_extensions → `inspect.signature` → not in the shadowing module → crash.
- **What to do differently:**
  - **Never put Python scripts in `/tmp`.** Put one-shot helpers in the repo root (`/opt/<project>/`) so `sys.path[0]` is a directory you control.
  - If you must run from `/tmp`, invoke with `PYTHONSAFEPATH=1` (Python 3.11+) or `python -I` to disable prepending the script dir, or use `cwd=/opt/<project>` in the launcher — BUT `sys.path[0]` is script-dir not cwd, so `cwd` alone doesn't help unless you also move the script.
  - Preventive rule: `pgrep -l "/tmp/.*\.py"` in health checks — flag any long-running Python process whose script lives in `/tmp`.

## 2026-04-12 SEC servicer-cert PDFs were rendering empty — image-based EDGAR filings
- **What went wrong:** After fixing the routing, tapping a servicer-certificate PDF showed what looked like a bare exhibit reference ("EX-99.1 2 crvna….htm EX-99.1") and no report data. The PDFs themselves were ~16KB — all "content" but no content.
- **Root cause:** Carvana's servicer certs on EDGAR are image-based filings: each page is a JPG (`crvna<deal>servicerrepo001.jpg`, `002.jpg`, …) referenced by relative URL inside the HTML, with a 1pt white hidden OCR text layer overlaid for accessibility. `generate_pdfs.py` fed the cached HTML straight to weasyprint, which had no way to resolve the JPG relative paths (we never downloaded them). Result: PDFs contained only the invisible OCR text plus the `<DOCUMENT>`/`<TYPE>` header — looked like an exhibit reference page.
- **What to do differently:**
  - For image-based filings, don't try to re-render to PDF. Either (a) download the JPG assets alongside the HTML and rewrite relative paths before rendering, or (b) link directly to the SEC EDGAR URL, which already serves the JPGs in place.
  - We chose (b) — simpler, always current, no cache/PDF pipeline to maintain. Builders creating "download the filing" UX should default to linking to sec.gov unless there's a specific reason to re-host.
  - When the HTML has tiny `font-size:1pt;color:white` text blocks, that's the OCR layer — the real content is the `<IMG>` siblings. Treat that as a signal to link out, not render locally.

## 2026-04-12 SEC filing links on live were 404'ing silently
- **What went wrong:** SEC filing PDF links on the live Carvana ABS dashboard returned the dashboard page instead of the PDF. User noticed because tapping them didn't open the document.
- **Root cause:** `promote_to_live()` in `carvana_abs/generate_preview.py` only copied `index.html` from preview/ to live/. The `docs/` tree (where generate_pdfs.py writes PDFs) was never synced. nginx `try_files $uri $uri/ /CarvanaLoanDashBoard/index.html` masked the failure — every missing PDF silently served the dashboard page with HTTP 200, so no monitoring caught it.
- **What to do differently:**
  - When promote is a "copy the preview" step, copy the whole preview directory (or an explicit rsync of every asset subdir), not just a curated list of files. Asset subdirs like docs/, images/, data/ are easy to forget.
  - Any nginx `try_files ... /index.html` fallback masks 404s on sibling assets. For URLs that are expected to be files (PDFs, JSON, CSV), tests must assert the response `content-type`, not just HTTP 200.
